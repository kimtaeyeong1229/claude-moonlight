# Moonlight v0.2 — 계획

> v0.1 의 내용을 그대로 계승하고, **설치/배포 방식**과 **번역 엔진 출처**를 명시한다.

## 한 줄 요약
영어를 몰라도 논문을 편하게 읽을 수 있게 해주는 **macOS 전용 앱**.
좌측에 논문 원문, 우측에 동일한 레이아웃의 **완전 한국어 번역본**을 나란히 보여주고,
한쪽 문장에 hover 하면 반대편의 대응 문장이 **파란색 형광펜 배경**으로 강조된다.

## 배경 / 참조
- 기존 프로젝트 **Moonlight (claude-web-papers-kr)** — arXiv 논문을 한국어 Medium 스타일 HTML로
  번역해 모아 읽는 로컬 웹 대시보드 — 를 참조해 출발한다.
  - 핵심 자산: PTY 기반 인터랙티브 `claude` 워커, 파일시스템 잡 큐(app ↔ worker 분리),
    PDF→읽기→번역→그림/표 추출→HTML 빌드 파이프라인, aiohttp 대시보드 + 프런트엔드.
- 이번 프로젝트는 "모아 읽기"가 아니라 **한 논문을 원문/번역 대조로 깊게 읽기**에 초점을 맞춘
  **맥북 전용 데스크톱 앱**이다.

## 목표 (v0.1 스코프 계승)
1. **2단 대조 뷰 (Side-by-side)**
   - 좌: 논문 원문 (영어 / 한자 포함 그대로)
   - 우: 좌측과 **동일한 레이아웃**의 완전 한국어 번역본
   - 양쪽이 문단·문장 단위로 정렬되어 같은 위치에 대응 내용이 오도록 한다.
2. **문장 단위 하이라이트 연동 (핵심 기능)**
   - 원문(영어 혹은 한자)의 한 문장에 마우스를 **hover** 하면,
     반대편(번역본)에서 **동일한 문장**의 배경이 **파란색 형광펜 색**으로 칠해진다.
   - 반대 방향(번역 → 원문)도 동일하게 동작한다.
   - 목적: 영어를 모르는 사람도 "이 한국어 문장이 원문 어디인지"를 즉시 눈으로 매칭.
3. **Figure / Table 보존**
   - 그림·표는 번역하지 않고 **원본 그대로 양쪽에 동일하게** 유지한다.
   - (캡션은 번역하되, 도표 이미지 자체는 원본 유지)
   - 좌우에서 같은 figure/table이 같은 위치에 오도록 정렬.

## 핵심 동작 원리 (구상)
- 번역 시 **문장 단위로 원문↔번역을 매핑**하는 정렬 정보를 함께 생성한다.
  - 각 문장에 공통 id(예: `s-12`)를 부여 → 좌우 DOM에서 같은 id를 가진 요소끼리 hover 연동.
  - figure/table에도 공통 id(예: `fig-3`, `tbl-1`)를 부여해 위치 정렬.
- 렌더링은 원문 HTML / 번역 HTML 두 벌을 만들고, 같은 id 기준으로 좌우에 배치.
- hover 하이라이트는 프런트엔드에서 id 매칭으로 처리(파란 형광펜 = 배경색 강조).

## ★ v0.2 신규 — 번역 방법 / 엔진은 참조 레포를 그대로 따른다
- **번역 파이프라인·엔진은 새로 만들지 않고 `claude-web-papers-kr` 의 방식을 그대로 계승**한다.
  - **PTY 인터랙티브 `claude` 워커**로 번역을 구동한다. **`claude -p`(헤드리스) 절대 금지** (참조 레포의 하드 제약).
    - `pty.openpty()` + `fork` + `execvpe`, 그리고 입력 제출은 **텍스트 → 짧은 대기 → 별도 `\r`**
      (claude TUI 의 붙여넣기+Enter 삼킴 버그 회피 — `worker.submit` 그대로).
  - 파이프라인 절차도 참조 레포의 `process_paper_prompt.md` 를 기반으로 한다:
    **PDF 다운로드 → 페이지 읽기 → 한국어 번역 → 그림/표 추출(`pdftoppm`+PIL crop) → HTML 빌드 → DONE/FAILED sentinel**.
  - 모델/품질도 참조 레포 기본을 따른다: `WORKER_MODEL=claude-opus-4-8`, `WORKER_ULTRACODE=1`(`/effort ultracode`),
    ultracode 시 번역 후 원문 대조 검증 1회.
  - 잡 큐(원자적 rename), 상태 파일 분리(`meta.json`/`stage.txt`/`tags.json`/`en_title.txt`) 컨벤션도 계승.
- **v0.2 에서 달라지는 부분은 "산출물 형식"뿐**: 단일 `paper.html` 대신
  **문장 단위 정렬 id 를 가진 원문/번역 2벌**(공통 `s-N`, `fig-N`, `tbl-N`)을 생성하도록
  `build_paper.py` / 프롬프트만 확장한다. (엔진 자체는 동일.)
- 자세한 코드/패턴은 `materials/01-reference-repo.md` 와 `materials/moonlight-reference/src/` 참조.

## ★ v0.2 신규 — 원클릭 설치 & 부팅 시 백그라운드 자동 실행 (macOS)

### 전제 (사용자 환경 가정)
사용자 맥북에 다음이 이미 설치·인증되어 있다고 **가정**한다:
- **Docker** (Docker Desktop 또는 동등 런타임) — 실행 중.
- **Claude CLI (`claude`)** — PATH 에 있고 로그인 완료 상태.
- macOS (PTY + launchd 사용).

### 설치 흐름 (사용자가 할 일은 딱 두 단계)
1. 레포를 **clone** 한다.
2. 에이전트(claude)에게 **"설치해줘"** 라고 말한다.

그러면 에이전트가 아래를 **자동으로** 수행한다 → 이후 **컴퓨터가 켜질 때마다 백그라운드로 항상 실행**.

### 에이전트가 수행할 설치 작업 (`install` 동작 정의)
설치를 요청받은 에이전트는 다음을 순서대로 실행한다. (멱등하게 — 다시 돌려도 안전하게 작성)
1. **사전 점검**: `docker info`(데몬 동작 확인), `which claude` + 로그인 상태 확인.
   둘 중 하나라도 없으면 명확한 안내 메시지를 내고 중단.
2. **빌드/준비**: 앱·워커를 담은 컨테이너 이미지를 빌드(또는 `docker compose build`).
   - claude CLI 인증/세션은 컨테이너에서도 쓸 수 있게 호스트의 claude 설정 디렉터리를 마운트하거나,
     PTY 워커가 호스트의 `claude` 를 호출하도록 구성(아래 "결정사항" 참조).
3. **부팅 자동 시작 등록**: macOS **LaunchAgent**(`~/Library/LaunchAgents/`)를 설치해
   로그인/재부팅 시 `docker compose up -d`(또는 컨테이너 start)가 자동 실행되게 한다.
   - 참조 레포의 `scripts/install-autostart.sh` 방식을 컨테이너 기동에 맞게 변형.
   - `RunAtLoad=true`, `KeepAlive=true`(죽으면 launchd 가 되살림).
4. **헬스 체크**: 컨테이너가 뜨고 대시보드 포트(기본 8090)가 응답하는지 확인 후 완료 보고.
5. **해제 경로 제공**: `uninstall` 시 LaunchAgent 제거 + 컨테이너 down.

### 산출물 (설치 자동화를 위해 만들 것)
- `Dockerfile` / `docker-compose.yml` — 앱(app) + 워커(worker) 서비스.
- `scripts/install.sh` (또는 에이전트용 `INSTALL.md` 지침) — 위 1~4 단계 자동화.
- `scripts/uninstall.sh`.
- macOS LaunchAgent plist 템플릿(부팅 시 `docker compose up -d`).
- 에이전트가 "설치해줘" 만으로 따라 할 수 있는 **명시적 설치 지침 문서**(claude 가 읽고 실행).

### 결정해야 할 사항 (착수 시 확정)
- **PTY 워커 ↔ Docker 관계**: 워커가
  (a) 컨테이너 안에서 호스트 claude 를 호출(소켓/마운트), 또는
  (b) 호스트에서 워커만 돌고 앱만 컨테이너 — 중 어느 쪽인지.
  → 참조 레포 제약(인터랙티브 PTY + 호스트 claude 로그인) 고려해 결정. 도커 안에서 PTY+claude 인증이
    매끄럽지 않으면 "앱=컨테이너 / 워커=호스트 LaunchAgent" 하이브리드가 현실적.
- 동시 처리 수(`WORKER_CONCURRENCY`) 기본값, 포트(기본 8090) 노출 방식.

## 결정해야 할 사항 (v0.1 에서 이월)
- 앱 형태: 네이티브(Swift/SwiftUI) vs 웹뷰 래핑(Electron/Tauri/Chrome app 모드) vs 로컬 서버+브라우저.
  - 기존 Moonlight 자산(파이썬 워커/파이프라인) 재사용 + Docker 배포를 고려하면 **로컬 서버 + 웹뷰** 가 유력.
- 문장 정렬 정밀도: 번역 단계에서 문장 매핑을 직접 생성할지, 사후 정렬할지.
- 한자(중국어/일본어 한문) 처리 범위 — 우선 영어 위주, 한자 hover 매칭은 동일 메커니즘으로 확장.
